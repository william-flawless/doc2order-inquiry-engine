# Day 2 Decisions

## Environment
- active environment for Day 2 setup: dev

## Event Ingestion Pattern
- S3 -> SQS -> Ingest Lambda

## Rationale
- decouples file upload from compute
- provides buffering and retry behavior
- improves reliability compared with direct S3 -> Lambda invocation
- supports future orchestration via Step Functions

## Existing Input Bucket
- bucket name: flawless-doc2order-emailpdf-prod
- rationale: reuse the same inbound bucket already used by the existing Doc2Order LLM processor
- note: environment and customer separation will be enforced at the S3 prefix level

## Evapo Input Prefixes
- Dev: input/dev-deterministic/evapo/
- Prod: input/prod-deterministic/evapo/

## Prefix Strategy Notes
- Evapo is using deterministic-specific landing prefixes
- the pipeline must not assume the generic pattern `input/{env}/{customer_id}/`
- environment and customer identification must support these explicit deterministic prefixes

## Idempotency Strategy
- each uploaded document will be assigned a deterministic `doc_id`
- `doc_id` is defined as a SHA-256 hash of:

  environment + bucket + key + etag + customer_id

## Why `doc_id` uses this structure
- `environment` prevents cross-environment collisions
- `bucket` identifies the source storage location
- `key` identifies the object path
- `etag` distinguishes changed content from the same object key
- `customer_id` ensures customer isolation and supports multi-customer scalability

## Example Raw Identity String
dev|flawless-doc2order-emailpdf-prod|input/dev-deterministic/evapo/sample.pdf|abc123etag|evapo

## Example `doc_id`
- generated by: sha256("dev|flawless-doc2order-emailpdf-prod|input/dev-deterministic/evapo/sample.pdf|abc123etag|evapo")
- stored as a lowercase hexadecimal string

## DynamoDB Table Purpose
- store one item per unique document
- prevent duplicate processing for the same file version
- provide a lightweight processing registry
